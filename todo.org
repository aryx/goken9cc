*** ic (compiler)

**** boolgen()/comrel() NaN-comparison bug -- deeper than expected
claude:
compilers/ic/cgen.c:763, compilers/vc/cgen.c:636,730 -- boolgen()
applies comrel[] (the integer logical-complement table, OLT<->OGE,
OLE<->OGT, OEQ<->ONE, etc) to the source-level relational operator
unconditionally, before checking whether the comparison is floating-
point. That complement is sound for integers (total order) but not for
IEEE-754 comparisons under NaN: e.g. "!(x < y)" is not equivalent to
"x >= y" when x or y is NaN (both are false for NaN operands). This is
shared kencc lineage code -- present in every backend's boolgen() (5c/
6c/7c/8c/vc/ic and the 5ck/7ck/8ck variants), but 7c/5c already carry a
compensating fix in gmove()/boolgen() ("ensure NaN comparison is
always false", substituting ABMI/ABPL/ABHI/ABLS for the naive signed
branches when typefd[et]). Confirmed real on both riscv and mips with
a live qemu run: "x = 0.0/0.0; y = 1.0;" wrongly evaluates x>y and
x>=y as true on riscv, x<y and x<=y as true on mips.

Tried the obvious fix (mirroring kencc-riscv's restructuring: move the
comrel[] application out from before the typefd check to only the
integer-specific branches) -- it does NOT work. Confirmed empirically:
it just changes *which* operators come out wrong (riscv: x<y, x<=y,
x==y wrong instead of x>y, x>=y -- a lateral move, not a fix).
Root cause is deeper: compilers/ic/txt.c's gopcode() (case OGE:1217,
OGT:1225, `cmpfloat:` 1246) has its own synthesis for OGE/OGT, because
RISC-V's FPU only has native FEQ/FLT/FLE, no FGT/FGE -- it computes
FLE/FLT then branches on the *inverted* sense (`ab`), and that
synthesis is independently NaN-unsound (assumes "!(x<y) implies x>=y",
the same unsound total-order assumption, just relocated). boolgen()'s
comrel[] turns out to be partly *compensating* for gopcode()'s
synthesis rather than purely redundant with it, which is why removing
it breaks cases it was accidentally getting right. A real fix needs to
untangle both pieces together (likely: fix gopcode()'s OGE/OGT
synthesis to be NaN-safe first, matching 7c/5c's ABMI/ABPL/ABHI/ABLS
approach as closely as this ISA allows, *then* revisit whether
boolgen()'s comrel[] still needs adjusting on top) -- not the
"easily testable" one-line reorder it first looked like

*** 6c (compiler)

**** OAS hardleft paths drop a nested assignment's value into nn
claude:
compilers/6c/cgen.c:114-117 (the "l->op==OINDEX && immconst(r)" fast
path inside the hardleft branch) does `gmove(r, l); break;` with no
`nn` propagation, and the general hardleft fallback at line 133
(`gmove(&nod, &nod1); regfree(...); break;`) has the same gap --
compare the non-hardleft path at lines 119-124, which correctly does
`if(nn != Z) gmove(r, nn);` right next to the equivalent store. So an
assignment-used-as-a-value through one of these two specific paths,
e.g. `x = (a[i] = 42);` where the OINDEX/immconst fast path or the
general hardleft fallback fires, would silently drop the outer
assignment (x keeps its old value) instead of receiving 42. 9front
adds the missing `if(nn != Z) gmove(...);` call to both spots. Real,
verbatim upstream-diverging logic bug -- but could not manufacture a
test that visibly misbehaves within budget (regalloc's nn-hint
mechanism seems to reliably honor nn whenever it's a plain free
register in every construction tried: array assignment, pointer-deref
assignment, under register pressure). Same epistemic status as several
already-landed 7c fixes this session. Independently present in
compilers/8c/cgen.c:~135,154 and compilers/8ck/cgen.c -- port to all
three together if landing this

**** fix OFUNC not stripping a leading comma from the callee
claude:
compilers/6c/cgen.c:882 is missing `l = uncomma(l);` at the top of the
OFUNC case, unlike vc, 8c, 8ck and 7c (all of which already have this
exact call, see e.g. compilers/vc/cgen.c:284) -- 6c is the odd one
out. A callee reached through a comma expression, e.g.
`(sideeffect(), fp)(args)`, would hit `diag(n, "bad function call")`
or miscompile. Same status as the already-landed vc fix (see
plan9front.txt's mips section): proven correct by 4 sibling backends
now, but a quick `(g(), fp)()`-style probe didn't visibly discriminate
within budget on 6c either

**** LATER add missing SSE3/SSE4.1 mnemonics
claude:
9front's 6a/include/6.out.h/6l support JMPF, MOVQL, HADDPD/HADDPS,
ADDSUBPD/ADDSUBPS, HSUBPD/HSUBPS, ROUNDSD/ROUNDSS/ROUNDPD/ROUNDPS,
DPPD/DPPS -- goken has none of these (no enum entries, no 6a mnemonic-
table rows, no linkers/6l/optab.c rows). Real missing feature, not a
bug fix; needs coordinated additions across three files. Note 9cc
itself is actually a superset of both (full AVX/VEX + Y-register
support) that neither 9front nor goken carry -- goken's 6a currently
matches plan9foundation's simpler baseline, a deliberate choice per
changes.txt, not an oversight; low priority

*** 6l (linker)

**** RELAX autoffset negative-frame clamp is incomplete
claude:
linkers/6l/pass.c:510-511 clamps the local `autoffset` to 0 when
negative, but not `p->to.offset` itself, which some downstream reads
(current-frame tracking, AADJSP sizing) still see as negative. 9front
also clamps `p->to.offset = 0`. Low-impact edge case, only reachable
via a hand-written negative `TEXT ...,$-N` frame directive -- not
something real C source produces

*** cross-cutting (applies to 6a/6c/6l too, not new findings)
claude:
several TODOs already tracked elsewhere (mips section above, and
Last1/Last2 below) were reconfirmed as applicable here too, extending
their scope rather than being new items: the growable regopt() NRGN
region array, the ".ret" register-tracking carve-out, sprint() ->
snprint() unbounded-buffer cleanup, and especially the nodrat
(struct-copy scratch temp) non-reentrancy bug -- compilers/6c/gc.h:147,
150,157 confirms 6c shares the exact same nodrat/nrathole/symrathole
global-temp pattern as every other backend (5c/5ck/7c/8c/8ck/ic/vc),
so this bug is genuinely tree-wide, not mips-specific as originally
scoped; worth raising its priority given how universal it is
