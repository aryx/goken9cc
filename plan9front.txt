Most of the code in goken9cc derives from code in Plan 9 4th edition
available at https://9p.io/sources/plan9/sys/src/ and
https://github.com/plan9foundation/plan9, as well as code from
https://code.google.com/p/ken-cc/ and https://github.com:golang/go/ at
the 99a10eff16b79cfb8ccf36e586532a40b17a203c commit point
(see pad.org for more info).

Some code and fixes comes also from 9front at https://github.com/9front/9front
such as:
 - 7a/7c/7l:
 - 6a/6c/6l:
 - va (mips): assemble() copied its filename argument into a fixed
   `char ofile[100]` with a raw strcpy() and no bounds check --
   confirmed this crashes ("*** stack smashing detected ***", SIGABRT)
   on a source path >=100 bytes. Ported 9front's fix: dynamically
   sized via strdup()/smprint() instead. See
   tests/s/regressions/mips_long_filename.s.tmpl.
 - va/vc/vl (mips): LL (load-linked) / SC (store-conditional) / SYNC
   (memory barrier) atomic instructions, needed for spinlocks/mutexes
   on MIPS -- ported from 9front's sys/src/cmd/v[al]/*, by mia soweli:
     10f1eeb7d v[al], libmach: add LL (load-linked) and SC (store-conditional)
     ca2589a11 v[al]: add SYNC (memory barrier)
     e7d099ac4 vl: LL operand order and scheduling
     2eb28f43e vl: atomic instruction scheduling
   See tests/s/features/mips_atomic_llsc.s.
 - vl (mips): linker instruction scheduler could hoist a stack-pointer
   -adjusting instruction above an earlier load in the same window,
   changing the load's effective address if reordered -- ported from
   9front's sys/src/cmd/vl/sched.c, by cinap_lenrek:
     40ed531364 vl: prevent stack altering instructions to be hoisted above loads
   (found via a libmemdraw bug on real 9front; no dedicated minimal
   test in this repo yet -- see linkers/vl/sched.c sched().)
 - vc (mips): register allocator's regopt() used a fixed
   Rgn region[600] array and gave up (warn + stop optimizing further,
   not a crash) past 600 candidate regions -- ported 9front's growable
   version (region starts at 300, doubles via allocn() as needed). No
   dedicated test: NVAR=160 caps distinct tracked variables, and
   reaching 600 *regions* needs enough control-flow complexity that a
   cheap synthetic generator (2000 statements, 700 distinct locals)
   still didn't trigger the old cap in testing.
 - vc (mips): mkvar() excluded every "."-prefixed pseudo-symbol from
   register-allocator tracking, including ".ret" (the function
   return-value pseudo-variable, see compilers/vc/txt.c ginit()) --
   ported 9front's carve-out so ".ret" specifically stays eligible for
   optimization like any other variable. Not a correctness bug, purely
   a missed optimization; no dedicated test (nothing to observably
   fail, just a quality difference in generated code that would need a
   golden-object diff to show, and this repo doesn't otherwise
   golden-diff vc's own register allocator's choices).
 - vc (mips): cgen()'s OFUNC case didn't strip a leading comma
   expression from the callee before checking it's a plain indirect
   call (l->op == OIND), e.g. a function pointer reached through
   `(sideeffect(), fp)(args)` -- so a complex-enough comma-wrapped
   callee hit "diag(n, "bad function call")" or miscompiled instead
   of working. Fixed with `l = uncomma(l);` (uncomma() already exists
   in shared cc/ code, and compilers/8ck/cgen.c, compilers/8c/cgen.c
   and compilers/7c/cgen.c all already have this exact call -- vc was
   the odd one out). No dedicated test: getting the compiler down this
   specific "complex indirect call through a comma expression" branch
   with a small snippet didn't work within this pass's budget (a
   `(g(), fp)()` -style test compiled to identical code with and
   without the fix, meaning it wasn't taking the intended path);
   applied on the strength of the fix being a verified no-op unless
   l->op == OCOMMA, and already proven correct by 3 sibling backends
   in this same repo.

Some fixes were found independently in both 9front and this repo's own
principia-softwarica lineage:
 - vc (mips): align()'s Aaut3 case ("total align of automatic") never
   set `w`, so stack "safe" spill-temp regions (regsalloc(), used e.g.
   when marshaling a call whose argument itself contains another call)
   were not rounded up to a MIPS word, skewing the whole function's
   frame size by a non-word amount. The 5c/7c (arm/arm64) backends
   already carried the fix (imported from principia-softwarica, itself
   independent of 9front); ported the same one-line fix to vc. 9front
   found and fixed the identical bug in their vc, independently:
     e8c8f5d82 vc: word align automatics
   See tests/c/regressions/mips_aut3_align.c.
